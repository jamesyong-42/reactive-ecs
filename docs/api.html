API reference

Every exported function, World member, scheduler, and type. For concepts and patterns, see the guide. All subscription methods return an Unsubscribe function; all observers fire synchronously inside the mutating call.

Definition#

defineComponent#

Defines a component type. defaults sets both the TypeScript shape and the initial values merged under data passed to addComponent. Data can be any shape — numbers, strings, arrays, nested objects, class instances. Names must be unique across all defineComponent calls in a world; a collision throws on first store access. Returns a frozen identity object.

defineTag#

Defines a zero-data boolean marker — presence is the information (Selected, Hovered, Dirty).

defineRelation#

Defines a relation — a managed, inverse-indexed, lifecycle-cleaned edge between two entities. See RelationOptions for exclusivity bounds and the onTargetDestroy policy. The returned object carries the normalized options (defaults applied).

defineResource#

Defines a world-level singleton (camera, config, a spatial-index instance). The store is created lazily from defaults on first access.

defineSystem#

Defines a named system. after / before (string or string[]) constrain ordering within a scheduler; phase assigns the system to a PhasedScheduler phase and is ignored by SystemScheduler. See SystemDef.

Not#

Wraps a component or tag as a negated query term: query(A, Not(B)) matches entities that have A and do not have B. A type whose store was never created counts as absent. Negated terms participate in incremental cache maintenance like positive ones.

createWorld#

Creates an independent world — entities, component/tag/relation stores, resources, query caches, and subscriptions are all per-world.

Ownership is enforced by freeze-on-write (RFC-007), always on, every build — there is no freeze option. Managed plain data entering a store (addComponent, updateComponent, resource instantiation, setResource) is deep-frozen in place — arrays and {}-objects at any depth, never class instances or typed arrays (those are borrowed by reference). So in-place mutation of a read throws in strict mode instead of silently bypassing change tracking, and the change journal's retained values are immutable by construction.

{ maxReentrancyDepth } (default 1000) bounds synchronous handler re-entrancy: a handler that mutates the world re-triggers handlers synchronously, so an unbounded handler→mutate→handler feedback loop would otherwise overflow the stack. Past this depth the world throws a loud cycle error instead.

tickWorld#

The sole frame-advance point — there is no other way to advance the clock. In order: runs fn(world) (your systems — typically (w) => scheduler.execute(w)), seals the open origin-run, resets the changes() window, delivers the sealed runs to onChanges handlers, fires onFrame, and increments currentTick. Delivery happens after the reset, so any mutation an onChanges (or onFrame) handler makes accumulates into the fresh window and is delivered on the next tick — never nested into this one.

The frame always advances. If a delivery handler throws, the error is collected and the sequence continues — every later handler still runs, the window still resets, the clock still moves — and the lone error (or an AggregateError over several) is rethrown only after the tick has fully advanced. A throwing handler can never freeze the tick counter or strand the world between frames.

Call with no fn to flush and advance a frame without running systems — tickWorld(world) is the "just deliver" path for an app with no scheduler loop. Throws if called during delivery or mid-mutation.

World · properties#

currentTick#

Current tick number; advanced by tickWorld (a public read — it cannot itself move the clock).

entityCount#

Number of live entities.

mutationOrigin#

Origin tag of the current mutation window — undefined outside any withOrigin window (the implicit "local" origin). Because every observer fires synchronously inside the mutating call, a handler reading this always sees the origin of exactly the mutation that fired it. Readable anywhere, not just in handlers.

World · entities#

createEntity#

Creates an entity and returns its id. Ids are allocated from a monotonic counter and never reused. Fires onEntityCreated. O(1).

createEntityWithId#

Creates an entity under a caller-chosen id — the restore primitive for id-preserving deserialization. On success the counter advances to id + 1 and onEntityCreated fires; observably identical to a createEntity() that landed on id.

  • Throws if id is not a positive integer, is already alive, or is below the internal counter — ids are never reused, so restore must create entities in ascending id order.
  • Restores into a fresh world only; it cannot resurrect a destroyed id in a live one. For undoable deletion prefer a tombstone tag (see document vs. interaction state in the guide).

setNextEntityId#

Restores the id counter exactly, so subsequent createEntity() calls allocate from n. The counter only moves forward — throws if n is not an integer or is below the current counter. Call after the restore loop: the saved counter can exceed the highest restored id (entities destroyed before the save consumed ids), and restoring it keeps stale references stale forever.

destroyEntity#

Destroys an entity, removing all its components, tags, and relation edges (as source or target). No-op if already dead. The observable sequence: onEntityDestroyed listeners (data still readable) → query caches drop the id → relation sweep (onRelationRemoved per edge, policy effects collected) → per-component onComponentRemoved (id appears in changes().removed(C)) → per-tag onTagRemoved (id appears in changes().removedTag(T)) → deferred onTargetDestroy effects apply (cascades, tags). Handlers fired during the sweep may read freely, but every mutating World method throws until the sweep completes — react instead from a later system over changes().removed(C) / changes().removedRelation(R), or from an onTargetDestroy policy. The guard lifts before the deferred policy effects apply (those are the world's own mutations), so cascades and { tag } policies work unchanged. See the guide for the full contract.

entityExists#

true while alive — and false forever after destroy. A stale id can never alias a new entity; no generational indices needed.

World · components#

addComponent#

Attach-or-replace (upsert). data is optional and partial — shallow-merged over the type's defaults; the merged value is deep-frozen in place (managed plain data at any depth; class instances and typed arrays kept by reference), immutable by construction. Entities may safely share a frozen subtree — it can only be replaced, never mutated. On first attach, observers fire with prev === undefined; on an entity that already has the component, the value is replaced and observers fire with prev = the old value — so prev === undefined reliably means "first attach". changes() records the net transition since the tick began: absent at tick start lands in changes().added, present at tick start lands in changes().changed. Throws if the entity is not alive. A full-value replace is just addComponent with the complete value.

removeComponent#

Detaches a component. Fires onComponentRemoved before the data is deleted (so prev is readable). changes() records the net transition: a component present when the tick began lands in changes().removed; one added earlier this tick nets to absent→absent and lands nowhere. No-op if absent.

getComponent#

Current value, typed read-only, or undefined if not present. The returned object is the live store value — managed plain data is deep-frozen on write, so mutating it in place throws in strict mode; write through updateComponent or addComponent.

hasComponent#

Presence check.

updateComponent#

Transforms an existing component with a pure recipe. prev is the live (frozen) value; return the next value — p => ({ ...p, x }) is the one-level merge, and you can compute from prev or branch. The returned value is frozen and stored. Returning prev unchanged is a free no-op: no write, no event, no changes() entry. Fires onComponentChanged with the previous value; the entity appears in changes().changed (or stays in changes().added if it was attached this tick). A write that throws (recipe, cycle, or accessor) leaves the kernel's stores and change tracking unchanged. Non-creating by design — absence is never silent:

World · tags#

addTag#

Adds a tag. Fires onTagAdded; appears in changes().addedTag when the tag was absent when the tick began (re-adding a tag held at tick start is a vacuous present→present — recorded nowhere). No-op if already present.

removeTag#

Removes a tag. Fires onTagRemoved; appears in changes().removedTag when the tag was present when the tick began (add-then-remove within one tick nets to nothing). No-op if absent.

hasTag#

Presence check.

World · relations#

relate#

Adds an edge from source to target.

  • Throws if source or target is not alive — a born-dangling edge is impossible.
  • No-op if the exact edge already exists.
  • Exclusivity replaces: if a bound would be violated, the existing edge is removed first — removed-then-added events fire, mirroring addComponent overwrite semantics.
  • Appears in changes().addedRelation when the edge was absent when the tick began — re-relating an edge unrelated earlier this tick is a vacuous present→present, recorded nowhere — and fires onRelationAdded.

unrelate#

Removes the edge to target — or, with target omitted, all of source's outgoing edges for this relation. No-op if absent. Appears in changes().removedRelation when the edge existed when the tick began (relate-then-unrelate within one tick nets to nothing) and fires onRelationRemoved per edge.

getTargets#

The targets source points at. [] if none.

getTarget#

Source's single target — convenience for sourceExclusive relations.

getSources#

The sources pointing at target — the always-coherent inverse index, maintained by the store, never by you. [] if none. Entity first, like every per-entity read: getSources(parent, ChildOf) reads "who points at parent?".

World · resources#

getResource#

Reads a singleton. The store is created lazily from the type's defaults on first access.

setResource#

Shallow-merges a partial update over the singleton. The merged value is frozen on write (managed plain data, deep), and the stored value is replaced, never mutated in place.

updateResource#

The functional counterpart to setResource — transforms the singleton with a recipe prev => next. The returned value is frozen and stored; returning prev unchanged is a no-op. Lazily creates the resource from its defaults on first touch, like getResource.

World · queries#

query() results are cached per type-combination and maintained incrementally as components, tags, and entities change — repeat calls do no scanning. Change detection for the tick is changes(): a value-carrying view by the net transition since the tick began — absent→present = added, present→present with ≥ 1 write = changed, present→absent = removed, absent→absent = nothing — disjoint by definition. See the partition rule.

changes#

The net change detection for the current tick, carrying values. A live view: the window accumulates as the tick runs, but each accessor call materializes a stable snapshot (every value is a zero-copy reference to an immutable store snapshot; only the container is fresh), so iterating one while the loop body mutates the world is safe, and a later call reflects later writes. Not retainable across a tick. One accessor per partition slot, each keyed by entity id and carrying the value:

removed(C) carries the window-start value (so applyChanges can restore pre-window state). The value of a component replaced mid-window is not here — net diffs cannot serve that; use the synchronous onComponentRemoved event for cleanup of every intermediate value.

query#

Entities matching all positive components/tags and none of the Not() terms. With no arguments, returns all live entities (a copy of the maintained alive set — cheap). Throws if all terms are negated: a pure-negative query would need its own cached set re-evaluated on every mutation of the negated types and on every entity create — that standing maintenance cost, not the one-off scan, is what the throw refuses. Keep at least one positive term.

disposeQuery#

Drops the cache entry and reverse-index registrations for this exact query signature (order-insensitive, like query). No-op if the signature was never queried. A later query with the same signature rebuilds the entry with one scan over the smallest candidate store — that rebuild is the only full-scan path in the library — and is incrementally maintained again afterwards.

queryTagged#

All entities with the tag.

queryRelation#

All live edges as [source, target] pairs.

World · events#

All handlers fire synchronously inside the mutating call, in subscription order; per-entity (or per-source) handlers fire before wildcard handlers. Every method returns an Unsubscribe function. The one exception is onChanges, the delivered channel, which fires at the tick boundary, not at mutation time.

Two channels

The synchronous events (onComponentChanged and friends) are for reacting — invariants, GPU/DOM cleanup, fine-grained value subscriptions: things that need the moment. onChanges is for recording — undo, sync, derived state, devtools: things that want the net result of a coherent unit of work, with attribution, after the work is done. Reach for the channel that matches the job.

onChanges#

Subscribe to delivered change detection. The handler fires once per sealed origin-run, in capture order, when the tick advances via tickWorld — once per tick in the common single-origin case, and once per contiguous same-origin run when origins interleave mid-tick (a withOrigin apply landing between local writes delivers [local, remote, local]). The world is final and coherent during delivery: handlers may read freely and may mutate — their writes apply immediately, but accumulate into the next window and are delivered on the next tick, never nested into this one. Delivery is serial and exactly-once: every handler receives every run even if an earlier handler throws (errors are collected and rethrown as an AggregateError after the drain, and the frame still advances — see tickWorld). Empty runs are not delivered.

DeliveredChanges extends WorldChanges — every added / changed / removed accessor, the same partition — and adds exactly one field:

origin is well-defined because a run is origin-homogeneous: undefined is the implicit local origin, never "mixed" — an origin change seals the run. Unlike the live changes() view, a delivered run is an immutable snapshot frozen at seal time and safe to retain — an undo stack may hold one across thousands of ticks (the ownership rule makes that zero-copy and sound). Filter on origin to suppress echo: an undo recorder records only origin === undefined, a sync outbox sends only origin !== REMOTE.

onComponentChanged#

Fires on every write — including the initial addComponent, where prev === undefined, so one subscription covers "appeared" and "updated". Pass entityId to observe a single entity — the shape a useComponent() hook needs.

onComponentRemoved#

Fires before the data is deleted — prev is the value being torn down, the right moment to release GPU buffers, DOM nodes, or subscriptions keyed by it. Also fires for each component an entity owned at the moment destroyEntity was called.

onTagAdded#

Fires when the tag is added; optional per-entity filter.

onTagRemoved#

Fires when the tag is removed — including during destroyEntity.

onRelationAdded#

Fires when an edge is added. A bare id filters by source (matching the entityId filter on component events); { target } filters by target — the parent-watches-children shape; { source, target } matches the exact edge. Emit order: per-source → per-target → wildcard.

onRelationRemoved#

Fires when an edge is removed — via unrelate, exclusivity replacement, or destroyEntity of either endpoint. During a destroy the dying entity's components and tags are still readable from the handler, but mutating the world mid-sweep throws: read freely, react from a later system over changes().removedRelation or an onTargetDestroy policy instead.

onResourceChanged#

Fires synchronously inside every setResource (after the merge) and updateResource (after the recipe) — prev is a top-level snapshot of the value before the write (the object itself never aliases next; nested values a merge didn't replace are shared — safe, since stored values are only ever replaced, never mutated in place), next is the live post-write value. No per-entity filter (resources are singletons). Subscribing lazily creates the store. Handlers can read mutationOrigin.

onEntityCreated#

Fires on createEntity and createEntityWithId.

onEntityDestroyed#

Fires first in the destroy sequence — the entity's components, tags, and relations are still fully readable, so callbacks can snapshot whatever they need.

onFrame#

The post-delivery flush hook: fires once per tickWorld, after onChanges delivery and the changes() window reset — the moment a UI adapter pushes its batched, already-collected state. It no longer reads this-tick change data (the window is already reset by the time it fires); for change data, use onChanges or, inside a system, changes(). A throwing onFrame handler is collected like an onChanges one — the frame still advances.

World · origin#

withOrigin#

Tags every mutation made synchronously inside fn with origin, readable by handlers via mutationOrigin — the echo-suppression primitive for modules that both observe and mutate (sync adapters, undo journals). The library attaches no semantics to any origin; the vocabulary is yours (symbols recommended).

  • Throws unless origin is a string or symbol — undefined stays unforgeable as "no origin".
  • Re-entrant: nested calls stack; innermost wins; exiting restores the enclosing origin — also on throw.
  • Returns fn's return value.
  • Synchronous only: if fn is async, mutations after the first await are not tagged — re-enter withOrigin in continuations.
  • destroyEntity cascades — teardown events and relation policy effects — inherit the origin active at the call site. changes() stays origin-blind; per-origin attribution rides on the delivered onChanges runs.

World · introspection#

All live entity ids.

getRegisteredComponents#

getRegisteredTags#

Every TagType that has had a store created in this world.

getRegisteredRelations#

Every RelationType that has had a store created in this world — the fourth member of the introspection group, the one that makes a complete user-land entity snapshot possible (lazily-registered relation types are otherwise unenumerable).

getRegisteredResources#

getComponentsOf#

getTagsOf#

World · change utilities#

applyChanges#

Replays a WorldChanges (or a delivered DeliveredChanges, which is one) — or its inverse — through the public write API. This is the ground support that makes app-level undo/redo a one-liner each way; it carries no privileged access. Pass { origin } and it wraps the replay in withOrigin for you, so an onChanges recorder filtering by origin ignores the replay:

Validate-first. With onMissing: 'throw' (default) it scans every change for a dead-entity reference and throws before mutating anything, so a partial apply can't corrupt the world — absence is never silent. 'skip' no-ops those entries and reports them in skipped (the multiplayer case: an op on an entity a peer deleted no-ops at that key).

Entity lifecycle is not replayed. Component, tag, relation, and resource changes are applied; created / destroyed entries are not — entity creation and destruction are app policy (tombstone, or an identity index for resurrection), and each such entry is reported in skipped. Removes and unrelates run before adds and relates, so exclusivity replacement applies cleanly.

World · frame lifecycle#

Frame advance is internal to tickWorld — the sole frame-advance point. It seals the open origin-run, resets the changes() window, delivers, fires onFrame, and increments currentTick as one indivisible step; there is no public way to advance the frame without delivering.

SystemScheduler#

Orders systems by after / before constraints using a stable topological sort (Kahn's algorithm; registration order breaks ties). Cycles throw with the names involved. A constraint naming a system that was never registered throws at the first execute() (or getSystemNames()) after any register/remove, naming both parties — a typo must not silently reorder the pipeline. Validation is deferred, so systems can be registered in any order. Optional-dependency ordering across app configurations is what phases are for.

register#

Adds a system. Re-registering a name replaces the existing system (hot-reload friendly) and invalidates the cached sort.

remove#

Removes a system by name.

execute#

Runs all systems in dependency order, bracketing each with the profiler's beginSystem / endSystem if attached.

getSystemNames#

The resolved execution order (triggers the sort).

size#

Number of registered systems.

profiler#

Optional instrumentation hook — see SystemProfiler.

PhasedScheduler#

Runs systems in a caller-defined phase order — the library ships zero phase opinions. Within a phase, after / before topo-sort identically to SystemScheduler (one runs internally per phase). Across phases, ordering is implicit in phase order; cross-phase constraints that contradict phase order are rejected at first execute() with an error explaining the fix (redundant ones — an after pointing at an earlier phase — are tolerated, since phase order already satisfies them). Constraints naming a system that is not registered in any phase throw at first execute(), same as SystemScheduler. The constructor throws on an empty or duplicate-containing phases array, or a defaultPhase not in it. Use as const on the phases literal for type-narrowed phase strings throughout. Also exposes remove(name), getSystemNames(), and profiler with the same shapes as SystemScheduler.

register#

Adds a system to the bucket for system.phase, falling back to defaultPhase. Throws if the system has no phase and no default is configured, or if the phase is not in the configured list. Re-registering into a different phase moves the system — it never runs twice.

getPhase#

The phase a system is registered in, or undefined.

getPhases#

The configured phase order (read-only view of the constructor input).

execute#

Runs one tick, phase by phase. Non-empty phases are bracketed with the profiler's optional beginPhase / endPhase; systems with beginSystem / endSystem. Cross-phase constraint validation happens lazily on first call and after every register/remove.

Devtools#

Two subpath exports; the core . entry is untouched. @jamesyong42/reactive-ecs/devtools is headless and dependency-free; @jamesyong42/reactive-ecs/devtools/react needs React (optional peer, ≥18). See the guide for the quick start.

createLifecycleRecorder#

Captures every entity's birth and death from the event hooks (not a poll), so entities born and destroyed within one tick are recorded. Attach before entities spawn. At death the entity's descriptor is frozen while its data is still readable (destroyEntity fires listeners before teardown). cap (default 4000) bounds memory by evicting the oldest dead records — never a living entity.

defaultDescriber#

Zero-config identity off composition: label from the first component's name (else first tag, else 'entity'), remaining tags as detail, default color, no outcome. The fallback when no custom describer is passed.

formatValue / formatComponentData#

The inspector's compact value formatting (x=10 y=20), exported for custom panels.

EcsInspector#

Draggable, resizable, layout-persisting floating window: live entity list (tags + component values at ~16 Hz), plus the timeline tab when a recorder is provided. Styles are injected once into a deduped <style> tag — no CSS import.

EntityTimeline#

Canvas-rendered waterfall of entity lifecycles — one bar per entity, birth to death (or the live "now" edge). Pan (drag / two-finger scroll), pinch-zoom the time axis, live-tail that re-engages at the right edge, wall-clock or tick units, hover tooltip. Bar colors come from the describer; outcome: 'win' | 'lose' draws green/red end-caps. Usable standalone or inside the inspector.

EntityDescriptor / EntityDescriber#

The one seam between the generic tools and your domain: identity read off composition. Pass a describer to get domain labels, colors, and outcome end-caps everywhere at once.

RelationOptions#

The two exclusivity bounds are independent: ChildOf is sourceExclusive only (one parent, many children); a true 1:1 ownership edge sets both. onTargetDestroy runs per source when the target dies, after the destroy sweep completes: 'clear' drops the edge, 'cascade' destroys the source too, { tag } adds the tag to the source.

NotTerm#

Negated query term from Not().

SystemDef#

runIf is evaluated immediately before the system would run, so it sees same-tick writes from earlier systems via changes(). On a skip the profiler's optional skipSystem fires instead of beginSystem/endSystem. Caveat: the changes() window resets at end of tick — order lazy readers after their writers.

SystemProfiler#

Attach to either scheduler's profiler property; the scheduler just calls the hooks if present.
