# RFC-002.2: Id-Preserving Entity Creation — the Minimal Restore Primitive

- **Status**: Implemented (v0.4.0 target) — maintainer counter-proposal to RFC-002 Request 2, accepted
- **Author**: James Yong
- **Date**: 2026-06-09
- **Supersedes**: RFC-002 Request 2 (world snapshot/restore). The motivation — infinite-canvas's hand-written remap pass at `serialization.ts:157-187` and the load-time dangling-ref risk on every new entity-referencing field — is incorporated by reference.
- **Verdict in one line**: Request 2 asked for a serialization subsystem; the only part user-land structurally cannot build is two tiny methods. Ship those; keep serialization in user-land where the opinions live.

---

## Summary

Two new `World` methods:

```ts
world.createEntityWithId(id: EntityId): EntityId   // create an entity with a caller-chosen id
world.setNextEntityId(n: number): void             // restore the id counter exactly
```

That is the entire request. With them, a consumer's existing serializer can restore a world **with every entity id unchanged** — and when ids are identical after load, every entity reference and every relation edge is still valid as-is. There is nothing to remap. The remap pass, and the class of bugs it exists to prevent, disappear.

```ts
// user-land restore — the consumer's own loop, no library serializer needed
for (const id of [...snapshot.entities].sort((a, b) => a - b)) {
  world.createEntityWithId(id);
}
world.setNextEntityId(snapshot.nextEntityId);
// then replay components / tags / relations through the normal public API —
// every stored EntityId still points at the right entity, untouched.
```

## Why this and not `world.snapshot()` / `restoreWorld()` / `world.import()`

RFC-002's own admissibility test — *"can the consumer build this correctly in user-land using the public `World` API?"* — cuts Request 2 into two very unequal parts:

**Serialization is buildable, and was built.** `getAllEntities()`, `getComponentsOf()` + `getComponent()`, `getTagsOf()`, `getRegisteredResources()` + `getResource()` walk everything; infinite-canvas's serializer exists and works. More importantly, *which* state is "document" versus "transient" — hover state out, spatial-index resource skipped, recognizers dropped — is an **application opinion**. RFC-002's proposed `SerializePolicy` / `Codec` / `Registry` / `serializeOverride` machinery is real surface area whose only job is to let the library carry an opinion that RFC-001 deliberately pushed to user-land. The same logic that kept bundles and FSMs out keeps codecs out.

**Restore is blocked by exactly one capability.** `createEntity()` always allocates a fresh monotonic id (`world.ts:277`), and `nextEntityId` is closed over inside `createWorld` (`world.ts:57`) with no setter. So a consumer restoring a document must renumber every entity, which forces hand-remapping every entity-typed field — the entire disease. Everything *else* about restore (iterating the snapshot, calling `addComponent`/`addTag`/`relate`) is plain public-API replay.

For honesty: id-preserving restore into a fresh world is *technically* expressible today by allocating ids `1..nextEntityId` and destroying the holes. After a long editing session `nextEntityId` can be orders of magnitude larger than the live entity count (ids are never reused), so this is O(counter) allocations plus an event storm — "cannot at acceptable cost," which is what the structural claim should have said. The conclusion stands either way.

So the library ships the missing capability and nothing around it. This also reverses RFC-002's Q6 sequencing answer: this RFC is days of work and deletes the driver's remap pass immediately — it can ship **before** relations (RFC-002.1), not after.

## Specification

### `createEntityWithId(id)`

- **Throws if `id` is already alive.**
- **Throws if `id < nextEntityId`** — i.e., the id region has already been consumed. This is the load-bearing guard: an id below the counter was either previously used and destroyed, or skipped; the world cannot tell which, and resurrecting a destroyed id would break the library's foundational guarantee that `entityExists(staleId)` is `false` forever (the guarantee that makes dangling-ref checks, RFC-001's Option-A removal semantics, and RFC-002.1's policy-free dangling safety all work without generational ids).
- **Throws if `id` is not a positive integer.**
- On success: marks `id` alive, advances `nextEntityId` to `id + 1`, fires `onEntityCreated` — in every observable way a normal `createEntity()` that happened to land on `id`.

The `id < nextEntityId` guard means restore must create entities in **ascending id order**. That is one `sort()` in the consumer's loop, documented, and it is what makes the never-reuse invariant unconditional rather than "unconditional unless someone restored carelessly."

### `setNextEntityId(n)`

- **Throws if `n < nextEntityId`** — the counter only moves forward, same invariant, same reason.
- Otherwise sets the counter, so subsequent `createEntity()` calls allocate from `n`.

Why this is needed at all, given that `createEntityWithId` already advances the counter past every restored id: **the saved counter can legitimately exceed `maxRestoredId + 1`.** Entities created and destroyed before the save consumed ids that no longer appear in the snapshot's entity list — but stale references to them may still be embedded in saved component data (a last-known-target id, an undo-history entry). If the restored world re-issued those ids to fresh entities, the stale references would silently resurrect and point at the wrong thing. Restoring the counter exactly preserves "stale forever" across the save/load boundary. Consumers should therefore persist `nextEntityId` alongside their entities and call this after the restore loop.

### Interaction: undo-of-destroy cannot resurrect ids — by design

The `id < nextEntityId` guard has a consequence worth stating explicitly, because the driver will hit it: **these methods enable id-preserving restore into a fresh world, not id resurrection in a live one.** Once an entity is destroyed, its id sits below the counter forever, and `createEntityWithId` will refuse it. An undo system that snapshotted an entity at destroy time (infinite-canvas's `snapshotEntity`/`rehydrateEntity` in `commands.ts`) cannot replay that snapshot under the original id in the same world.

This is not an oversight to be patched with an escape hatch. Resurrecting a consumed id would break the library's foundational guarantee — `entityExists(staleId)` is `false` forever — which is what makes dangling-ref checks sound, what RFC-002.1's policy-free dangling safety leans on, and what this RFC's own `setNextEntityId` rationale defends across the save/load boundary. A resurrected id would make every reference captured before the destroy (in component data, in *other* undo entries, in a peer's unsynced edit) silently point at the "new" entity, which may or may not be the same logical object. The current driver already exhibits the bug class in miniature: `ConsumeCommand` tracks a mutating `currentChildId` because rehydration changes the id, and any earlier undo entry that captured the old id (a `MoveCommand`'s `before` map) targets a dead entity after an undo-of-destroy.

Two supported user-land patterns, in order of preference:

1. **Tombstone instead of destroy (recommended).** Editor-level "delete" adds a `Deleted` tag and removes the entity from queries/serialization by convention; `destroyEntity` is reserved for true eviction (e.g. when the entity ages out of the undo horizon, at which point no history entry can reference it). Undo of a delete is then `removeTag(e, Deleted)` — the id never changed, every captured reference stays valid, and the change is observable through the ordinary tag channels. This is the shipped-product pattern (Excalidraw's `isDeleted`, tldraw's server-side tombstones, Yjs's internal tombstones), and it is also the CRDT-compatible one: a peer's concurrent edit to a deleted entity merges cleanly instead of dangling.
2. **Fresh-id rehydration with journal remapping (fallback).** If true destruction must be undoable, rehydrate under a fresh `createEntity()` id and have the history layer rewrite subsequent entries through its own `old → new` map. This is mechanical *for the journal* — it observes every mutation, so it knows every captured id — but it is exactly the remap discipline this RFC exists to delete from serialization, so treat it as the fallback, not the default.

The existing "throws on id below counter" test regression-locks this boundary; no new surface is implied.

### What this deliberately does *not* control

**Event silence.** RFC-002 proposed an `emit: false` mode so document-open doesn't storm observers. Under this design that's not a library switch — it's the consumer's sequencing, which they already control: restore into the world *before* attaching React subscribers and external indexes, then prime those indexes once from the restored state, exactly as on first boot. A consumer that must restore into a world with live subscriptions simply replays through the normal API and accepts the events — which is RFC-002's `emit: true` path, also already expressible. The library adds no mode.

## Implementation sketch

A handful of lines in `world.ts`, next to `createEntity` (`world.ts:276`):

```ts
createEntityWithId(id: EntityId): EntityId {
  if (!Number.isInteger(id) || id < 1) {
    throw new Error(`createEntityWithId(${id}): id must be a positive integer`);
  }
  if (alive.has(id)) {
    throw new Error(`createEntityWithId(${id}): id is already alive`);
  }
  if (id < nextEntityId) {
    throw new Error(
      `createEntityWithId(${id}): id is below the counter (${nextEntityId}); ` +
        `ids are never reused — restore entities in ascending order`,
    );
  }
  nextEntityId = id + 1;
  alive.add(id);
  for (const listener of createListeners) listener(id);
  return id;
},

setNextEntityId(n: number): void {
  if (!Number.isInteger(n) || n < nextEntityId) {
    throw new Error(
      `setNextEntityId(${n}): counter only moves forward (currently ${nextEntityId})`,
    );
  }
  nextEntityId = n;
},
```

No storage changes, no new stores, no behavior change on any existing path.

## What the driver deletes on adoption

- The remap pass (`serialization.ts:157-187`) — `ParentFrame`, `Children`, `ContainerChildren` field rewriting, gone.
- The "every new entity-referencing field must be registered for remap" maintenance rule and its dangling-ref failure mode.
- Once RFC-002.1 relations are adopted, relation edges round-trip through the same user-land replay (`relate(src, R, tgt)` with original ids) — the framework-side auto-remap RFC-002 wanted becomes unnecessary because there is nothing to remap.

The consumer's serializer itself stays — about as long as it is today, minus its hardest part.

---

## Non-goals

1. **`world.snapshot()` / `restoreWorld()` in the library.** Revisit only if a second consumer rebuilds materially the same serializer — that would be evidence the walk-and-replay boilerplate is worth owning. One consumer with working code is not.
2. **`SerializePolicy` / `Codec` / `Registry` / per-call overrides.** Application opinion; user-land.
3. **`world.import()` — merge-with-remap for copy/paste and multiplayer.** Genuinely useful, but *buildable in user-land today*: allocate fresh entities with plain `createEntity()`, keep your own `old→new` map, rewrite refs as you replay. The framework-side version is only compelling once relations exist and carry most references; if so it returns as a small follow-up ("replay this edge list through an id map"), not as part of this RFC.
4. **Snapshot format, versioning, determinism (RFC-002 Q5), binary encoding.** All properties of the consumer's format now.
5. **Bulk restore helper (`world.restoreEntities(ids)`).** Sugar over a three-line loop; skip until someone measures the loop mattering.

## Tests

- `createEntityWithId(5)` on a fresh world: entity 5 alive, `entityCount` 1, next `createEntity()` returns 6.
- Throws on: already-alive id; id below counter (create 5, then attempt 3); zero / negative / non-integer id.
- Fires `onEntityCreated` with the chosen id.
- `setNextEntityId` forward works; backward throws; restored counter prevents reuse (create 1-3, destroy 2, "save" counter 4, fresh world, restore 1 and 3 ascending, `setNextEntityId(4)`, next `createEntity()` is 4 — never 2).
- Round-trip property test: serialize a randomly mutated world with the introspection API, restore via ascending `createEntityWithId` + replay, assert every component/tag/entity-ref bit-identical with **zero remapping**.

## Versioning & sequencing

Strictly additive; ships as a minor. Small enough to ship immediately and independently — recommended order is **this RFC first** (instant payoff: the remap pass dies), then RFC-002.1 relations (the larger architectural lever), inverting RFC-002's Q6.
