# RFC-002.1: First-Class Relations — Minimal Subset

- **Status**: Implemented (v0.5.0 target) — maintainer counter-proposal to RFC-002 Request 1, accepted
- **Author**: James Yong
- **Date**: 2026-06-09
- **Supersedes**: RFC-002 Request 1 (the full relations proposal). RFC-002's motivation section — the edge census, the `ParentFrame`/`ContainerChildren` dual-source-of-truth, the ~12 `entityExists` guards — is incorporated by reference and not repeated here.
- **Verdict in one line**: relations are accepted as a concept; three pieces of the proposed surface are cut because RFC-002's own evidence shows they aren't needed for parity.

---

## Summary

reactive-ecs gains one new kind, the **relation**: a managed, inverse-indexed, lifecycle-cleaned edge between two entities.

```ts
const ChildOf = defineRelation('ChildOf', {
  sourceExclusive: true,        // ≤ 1 target per source (default: false)
  targetExclusive: false,       // ≤ 1 source per target (default: false)
  onTargetDestroy: 'clear',     // 'cascade' | 'clear' | { tag: TagType }  (default: 'clear')
});

world.relate(child, ChildOf, parent);
world.getTargets(child, ChildOf);     // → [parent]
world.getSources(ChildOf, parent);    // → all children — the always-coherent inverse
world.destroyEntity(parent);          // every child's edge cleaned, policy applied, removal observable
```

The store guarantees: **no relation edge ever survives the destruction of either endpoint**, removals (including destroy-driven ones) are observable through the same two channels every other primitive uses (per-tick batch + synchronous observer), and the reverse index is always coherent because the consumer never maintains it.

## What changed from RFC-002, and why

RFC-002 Request 1 proposed payload, a `'keep'` policy, and a custom-function policy with a deferred-`Effect[]` vocabulary and read-only `PolicyContext`. All three are cut. None of these cuts loses a capability the driver currently has — each cut is justified by RFC-002's own text.

### Cut 1: the custom policy function (and with it `Effect`, `PolicyContext`, and Q8)

RFC-002 called the escape hatch "required, not optional," citing the `Watches` nuance: a `Pending` recognizer may outlive its watched pointer, a non-`Pending` one must be cancelled. But look at what implements that nuance **today**: `recognizerIntegritySystem`, a *system that runs every tick and scans*. The replacement under enum-only policies is:

```ts
// onTargetDestroy: 'clear' on Watches — then, in the existing integrity system:
for (const [recognizer] of world.queryRelationRemoved(Watches)) {
  if (!world.entityExists(recognizer)) continue;
  if (!world.hasTag(recognizer, Pending)) world.addTag(recognizer, Cancelled);
}
```

Same per-tick latency the hand-rolled scan has now, but driven by an indexed signal instead of scanning every recognizer. The escape hatch buys *synchronous-within-destroy* semantics that the driver has never had and nowhere asks for. Cutting it deletes the `Effect` type, the `PolicyContext` facade, the "policies must not close over a world" rule, and an open question — roughly a third of the original spec.

If a future consumer demonstrates a policy that genuinely cannot tolerate one tick of latency, the function form can be added back compatibly (the option type widens).

### Cut 2: the `'keep'` policy

`'keep'` was the one policy that breaks the headline invariant ("no stale edge survives a destroy"), and it needed a five-row table plus a snapshot special case to specify. RFC-002 itself conceded the alternative. The `Follower` use case ("retain the last-known target id during the fade-out") is fully covered without it:

```ts
// onTargetDestroy: 'clear' on FollowTarget — capture the id at removal time:
world.onRelationRemoved(FollowTarget, (src, tgt) => {
  if (world.entityExists(src)) world.setComponent(src, Follower, { lastTarget: tgt });
});
```

Synchronous, zero information loss, and the invariant stays unconditional — which makes it worth stating in the README.

### Cut 3: per-edge payload

Three reasons:

1. **It was spec-incomplete as proposed.** `setPayload` populated no changed buffer — payload mutations would have been invisible to per-tick systems, silently breaking the symmetry promise this library just finished establishing in RFC-001. Doing payload *right* means `queryRelationChanged` and dirty-tracking for edges, a real design pass.
2. The only payload in RFC-002's census is `Follower.tau`, which works fine as a component field on the source.
3. It introduces a new storage idiom (string edge keys mapping to value bags) for one field.

Payload can return in a follow-up RFC with proper change-buffer semantics if a second real case appears.

Also trimmed as not-yet-needed conveniences: `related(s, R, t)` (use `getTargets(s, R).includes(t)`), `getRelationsOf(entity)` (devtools introspection — add when a devtool exists), `getPayload`/`setPayload` (no payload).

## API

### Definition

```ts
interface RelationOptions {
  sourceExclusive?: boolean;                          // default false — ≤1 target per source
  targetExclusive?: boolean;                          // default false — ≤1 source per target
  onTargetDestroy?: 'cascade' | 'clear' | { tag: TagType };  // default 'clear'
}

defineRelation(name: string, opts?: RelationOptions): RelationType
```

Returns a frozen identity object with `__kind: 'relation'`, exactly like `defineComponent`/`defineTag`/`defineResource` (`define.ts`). Name-collision detection on first store access mirrors `getComponentStore` (`world.ts:164`).

The two exclusivity bounds are independent. `ChildOf` is `sourceExclusive` only (one parent, many children). An ownership edge like `Dragging(pointer → widget)` sets both — a true 1:1, so two pointers cannot seize one widget.

The target→sources inverse index is **always maintained**; there is no option to disable it. Target-destroy cleanup requires it, and "give me the sources" is the whole reason to use a relation instead of a plain `EntityId` field.

### World methods

```ts
// mutation — mirror addComponent / removeComponent
world.relate(source, ChildOf, target): void        // throws if source or target is not alive; no-op if edge exists
world.unrelate(source, ChildOf, target?): void     // target omitted → remove all of source's outgoing edges; no-op if absent

// reads
world.getTargets(source, ChildOf): EntityId[]      // [] if none
world.getTarget(source, ChildOf): EntityId | undefined   // convenience for sourceExclusive relations
world.getSources(ChildOf, target): EntityId[]      // the inverse — always coherent

// per-tick batches — mirror queryAdded / queryRemoved; cleared by clearDirty()
world.queryRelation(ChildOf): [EntityId, EntityId][]         // all live edges (mirror of queryTagged)
world.queryRelationAdded(ChildOf): [EntityId, EntityId][]
world.queryRelationRemoved(ChildOf): [EntityId, EntityId][]  // INCLUDES destroy-driven removals (Option A, per RFC-001)

// synchronous observers — mirror onTagAdded / onComponentRemoved
world.onRelationAdded(ChildOf, (source, target) => void, sourceId?): Unsubscribe
world.onRelationRemoved(ChildOf, (source, target) => void, sourceId?): Unsubscribe
```

Eleven methods plus one `define*`. Each one is the relation-shaped mirror of something that already exists; the optional `sourceId` filter on observers matches the `entityId?` filter on `onComponentChanged`.

## Semantics

### `relate`

- Throws if source **or** target is not alive (mirrors `addComponent`, `world.ts:326-330`). A born-dangling edge is impossible. (RFC-002 Q1: resolved as recommended.)
- No-op if the exact edge already exists (mirrors `addTag`'s has-guard).
- **Exclusivity violation replaces.** If `sourceExclusive` and the source already points at `t1`, relating to `t2` unrelates `(source, t1)` then relates `(source, t2)` — removed-then-added events, mirroring `setComponent` overwrite semantics. Symmetric for `targetExclusive`. (RFC-002 Q3: resolved as recommended; a throwing `relateStrict` can follow on demand.)
- Populates the `added` buffer and deletes the edge from `removed` (net-cancellation: unrelate-then-relate of the same edge in one tick nets to nothing, per RFC-001's rule).
- Fires `onRelationAdded` (per-source handlers first, then wildcard — the established emit ordering).

### `unrelate`

Mirror of `removeComponent`: removes the edge from both indexes, populates `removed`, deletes from `added` (net-cancellation), fires `onRelationRemoved`.

### Entity destruction

This is the part user-land cannot guarantee, and the reason relations live in the library. Two roles, two behaviors:

**Entity dies as a source** — its outgoing edges simply vanish, symmetric to component teardown. `onRelationRemoved` fires per edge; `queryRelationRemoved` includes them. No policy applies; the source is gone.

**Entity dies as a target** — for every source pointing at it, the edge is removed (events + buffer as above) and the relation's `onTargetDestroy` policy contributes a deferred effect:

| Policy | Effect on each source | RFC-002's motivating case |
|---|---|---|
| `'clear'` *(default)* | none — edge dropped, source lives on | `ParentFrame` removed → child returns to root |
| `'cascade'` | source is destroyed too | selection chrome reaped with its target |
| `{ tag: T }` | tag `T` added to the source | recognizer cancelled on `Capture` loss |

(RFC-002 Q2: `'clear'` as default, resolved as recommended.)

**Effects are deferred — never applied mid-sweep.** `destroyEntity` removes the entity from `alive` before tearing down stores, so mutating during the sweep would observe a half-destroyed world. Cascade-destroys and tag-adds are collected during the sweep and applied strictly after every relation store has been processed:

- a `destroy` effect on an already-dead entity is a no-op (existing `alive` guard, `world.ts:284`);
- a `tag` effect on an entity that a *prior* effect destroyed is skipped (checked against `alive` at apply time);
- cascade chains terminate: each recursive `destroyEntity` only ever destroys live entities, ids are monotonic and never reused (`world.ts:57,277`), and re-destroying is a no-op — so mutual or cyclic cascade edges cannot loop.

**Sweep ordering — a deliberate correction to RFC-002's sketch.** RFC-002 appended the relation sweep *after* the component/tag teardown loops, which would mean `onRelationRemoved` handlers fire when the dying entity's components are already gone. That contradicts this library's established contract that destroy-time callbacks can still read data (`destroyListeners` fire before teardown precisely for this reason, `world.ts:285-287`). The relation sweep therefore runs **before** the component and tag loops:

```
1. destroyListeners fire           (entity fully intact — existing)
2. alive.delete(id)                (existing)
3. remove from query caches        (existing)
4. RELATION SWEEP                  (new — components/tags still readable from handlers)
     for each relation store:
       id as source → remove each outgoing edge: emit onRelationRemoved, buffer it
       id as target → remove each incoming edge: emit onRelationRemoved, buffer it,
                      collect the policy's deferred effect for that source
5. component teardown loop         (existing, RFC-001 shape)
6. tag teardown loop               (existing, RFC-001 shape)
7. apply deferred effects          (new — only now is mutation safe)
```

**Handlers must not mutate during the destroy sweep.** `onRelationRemoved` handlers invoked from step 4 run mid-destroy; calling `relate`/`destroyEntity`/`addComponent` from one is undefined behavior, the same implicit contract `onComponentRemoved` already carries during step 5. Read freely; mutate from a system reacting to `queryRelationRemoved` instead. (This is documented, not enforced — consistent with the existing observers.)

### Dangling-id safety

Unchanged from RFC-001's foundation: ids are monotonic and never reused, so any stale id a consumer copied out of a removal event stays `entityExists() === false` forever. No generational ids needed.

## Implementation sketch

A `RelationStore` joins `components` and `tags` as a third top-level map inside `createWorld`:

```ts
interface RelationStore {
  type: RelationType;
  forward: Map<EntityId, Set<EntityId>>;   // source → targets  (sourceExclusive ⇒ size ≤ 1)
  inverse: Map<EntityId, Set<EntityId>>;   // target → sources  (targetExclusive ⇒ size ≤ 1)
  added: Set<string>;                      // edge keys `${src}\0${tgt}` added this tick
  removed: Set<string>;                    // edge keys removed this tick (incl. destroy-driven)
  addedHandlers: Map<EntityId | '*', Set<RelationHandler>>;
  removedHandlers: Map<EntityId | '*', Set<RelationHandler>>;
}
const relations = new Map<string, RelationStore>();
```

Relations are a **side index, never an archetype/query key** — `query()` and the query cache are untouched. (RFC-002's reasoning stands: per-frame re-derived edges must not thrash structural buffers. The litmus for consumers: *make it a relation only if you query the inverse.*)

`clearDirty()` (`world.ts:637`) gains one loop:

```ts
for (const store of relations.values()) { store.added.clear(); store.removed.clear(); }
```

`queryRelationAdded`/`queryRelationRemoved` split the buffered keys back into numeric pairs on read; `queryRelation` walks `forward`.

Zero cost when no relation is defined: every new loop iterates an empty map.

## Non-goals

Carried over from RFC-002 (still excluded): pair-as-archetype-key / type queries; wildcards, transitivity, traversal (`up`/`cascade` walks — user-land on `getTargets`); hyperedges; relation data as queryable columns; modeling per-tick derived refs (`PointerTarget`, `PointerInside`) as relations.

Newly excluded by this RFC, all re-addable compatibly if evidence appears:

1. **Per-edge payload** — returns only with full change-buffer semantics (`queryRelationChanged`, dirty tracking). Its strongest known future case is CRDT replication of hierarchy: Figma fuses `(parentId, fractionalIndex)` into one atomic LWW property so a reparent and a reorder land together, whereas the payload-free shape models them as two registers (a `ChildOf` edge plus an order component on the child) that can briefly diverge under concurrent edits. The failure mode is cosmetic and self-heals, so this stays cut — revisit if concurrent reparent+reorder produces visible glitches in a synced consumer.
2. **`'keep'` policy** — the `onRelationRemoved`-copy pattern covers retention; the invariant stays unconditional.
3. **Custom policy functions** — returns only when a consumer demonstrates a policy that cannot tolerate one tick of latency.
4. **`related()`, `getRelationsOf()`** — conveniences; add with a devtools story.

## Acceptance tests (adoption gate, carried from RFC-002)

Prototype-first, as RFC-001 was. Before the surface is declared stable, the driver must demonstrate:

1. **Playground prototype**: `recognizerIntegritySystem`'s per-recognizer scans replaced by `queryRelationRemoved(Watches)` / `queryRelationRemoved(Capture)` reads — *including the `Pending` nuance, in user-land, without the custom-policy escape hatch*. This doubles as the empirical test of Cut 1.
2. **Production `ParentFrame` path**: one `ChildOf` relation deletes the materialized `Children`/`ContainerChildren` inverse and its apply/revert/load sync, and closes the `parentFrameActive` removal gap via `queryRelationRemoved(ChildOf)`.

If the minimal subset cannot do both, the cuts were wrong — escalate back to the RFC-002 shape (or withdraw, per RFC-002's own condition) rather than patching ad hoc.

## Migration & versioning

- Strictly additive: one new `define*`, eleven new `World` methods, new internal store, two new loops in `destroyEntity`/`clearDirty` that are empty no-ops for existing consumers.
- Ships as a minor (`0.5.0`).
- Tests mirror RFC-001's table: every `queryAdded`/`queryRemoved`/net-cancellation/destroy-inclusion test gets a relation twin, plus: exclusivity replacement emits removed-then-added; cascade chain terminates on a cycle; `{ tag }` effect skipped when a prior cascade killed the source; `onRelationRemoved` during destroy can still read the dying entity's components (the ordering guarantee).
