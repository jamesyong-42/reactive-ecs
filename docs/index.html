A reactive ECS
built for UIs.

reactive-ecs is a dependency-free TypeScript Entity-Component-System designed for UI frameworks, editors, and tools — not just game loops. Component changes, tag flips, and resource updates all emit events you can subscribe to. Queries are cached and update incrementally. A topological scheduler keeps systems ordered.

zero dependencies typescript-first esm + cjs browser · node · bun · deno

Introduction#

Most ECS libraries are built for throughput: pack data flat, scan it fast, and leave change detection to you. That is the right trade for a game simulating 100k entities. It is the wrong trade for an editor, a canvas tool, or any app where a UI must react to world state.

reactive-ecs makes the opposite trade. The library owns component storage, and because it owns the store, it can make guarantees a bring-your-own-storage toolkit cannot:

  • Every write through the world API emits. addComponent / setComponent / removeComponent all notify observers — no set() ceremony to forget. The flip side of the contract: reads hand you the stored object, so treat it as read-only and write through the API.
  • Subscriptions can target a single entity. onComponentChanged(Position, handler, entityId) is exactly the shape a useComponent() hook needs.
  • Removal is observable with the data still readable. onComponentRemoved fires before deletion, so handlers can release GPU buffers, DOM nodes, or subscriptions keyed by the dying value.
  • Entity ids never lie. Ids are monotonic and never reused — entityExists(staleId) is false forever.
  • Edges are managed. Relations keep an always-coherent inverse index, and no edge ever survives the destruction of either endpoint.
Positioning

If you want raw per-frame throughput for tens of thousands of game entities, use bitECS or becsy. If you want structured world state that a UI can subscribe to without reinventing the plumbing, this is for you. See non-goals for the full list of trade-offs.

Installation#

Ships ESM and CJS with bundled type declarations. No runtime dependencies. Requires Node ≥ 20 when used server-side.

Quick start#

The four calls in tick() are the whole frame protocol — run systems, notify frame subscribers, reset per-tick change buffers, advance the tick counter. Drive it from requestAnimationFrame, a setInterval, or only when something actually changed; the library has no opinion.

World & the tick#

A world is the container for all ECS state: entities, component stores, tag stores, resources, query caches, and subscriptions. Worlds are independent — create as many as you need.

Frame lifecycle#

Three methods make up the end-of-frame protocol. Call them in this order after your systems run:

MethodWhat it does
emitFrame()Fires every onFrame subscriber. The end-of-tick hook — flush batched UI updates here.
clearDirty()Resets the per-tick buffers behind queryChanged / queryAdded / queryRemoved / queryAddedTag / queryRemovedTag.
incrementTick()Advances currentTick by one.

Or let the helper encode the order: tickWorld(world, (w) => scheduler.execute(w)) runs your callback and then the three calls above — equivalent, harder to get wrong.

Note

Per-tick buffers accumulate until you call clearDirty(). If you skip frames (e.g. only ticking on input), the buffers simply hold everything that changed since the last clear — often exactly what you want.

Entities#

Entities are opaque integer ids (EntityId). Creation is O(1); destruction does work proportional to what the entity owns (components, tags, relation edges) plus the number of registered stores and query caches — fine at this library's scale, stated for honesty.

Ids are never reused#

Ids are allocated from a monotonic counter and never recycled. Once an entity is destroyed, its id is dead for the lifetime of the world.

Why this matters

Many ECS libraries recycle ids immediately, which means a stale id held by your UI, an undo stack, or a network message can silently alias a brand-new entity. Here a dangling reference is always detectably dangling: entityExists(staleId) returns false, never a lie. No generational indices required.

Components#

Components are typed data attached to entities. Define one with a name and a defaults object — the defaults double as the component's TypeScript shape.

Reading and writing#

  • Defaults merge on add. addComponent shallow-merges your data over the defined defaults, so partial initialization is safe.
  • Defensive clone on add. Plain objects and arrays in the initial data are deep-cloned at any depth, so two entities never share nested state by accident. Class instances are kept by reference, deliberately: the library never clones what it can't reconstruct. (Initial data must be acyclic.)
  • Shallow merge on set. setComponent takes a Partial<T> and merges it over the current value — one level deep — and emits a change event. Incoming plain data is defensively cloned here too, so a reference you keep to the data you passed can never mutate world state behind the API.

One level deep means nested objects are replaced wholesale, not merged into — and Partial<T> is partial at the top level only, so a nested object must be passed complete:

The contract

All writes go through addComponent / setComponent / replaceComponent / removeComponent, which is what lets the library guarantee that every change is queryable via change tracking and observable via events. Every write path defensively clones incoming plain data, so a reference you hold to data you passed in can never mutate world state behind the API. The other direction is convention: don't mutate the object returned by getComponent in place — the world can't see that. The Readonly typing on reads and event payloads enforces this at the top level at compile time; it's friction, not runtime immutability.

Tags#

Tags are zero-data boolean markers — presence is the information. Use them for states like Selected, Hovered, Dirty.

Tags participate in query() alongside components, have their own add/remove events, and their own per-tick buffers (queryAddedTag / queryRemovedTag).

Relations#

A relation is a managed, inverse-indexed, lifecycle-cleaned edge between two entities — for hierarchies, ownership, targeting, or any entity-to-entity reference you would otherwise store as a raw EntityId field and guard by hand.

The inverse index is maintained by the store, never by you — no more materialized children arrays kept in sync by hand. The litmus for when to use a relation instead of a plain id field: make it a relation only if you query the inverse.

  • Born-dangling edges are impossible — relate throws if either endpoint is not alive.
  • Exclusivity replaces. On a sourceExclusive relation, relating to a new target unrelates the old one first — observable as removed-then-added, mirroring setComponent overwrite semantics.
  • Re-relating an existing edge is a no-op, mirroring addTag.

Destroy policies#

When a target dies, every edge pointing at it is removed and the relation's onTargetDestroy policy is applied to each source. Policy effects are deferred until the destroy completes, so handlers never observe a half-destroyed world:

PolicyEffect on each sourceTypical use
'clear' (default)none — the edge is dropped, the source lives onparent removed → child returns to root
'cascade'the source is destroyed tooselection chrome reaped with its target
{ tag: T }tag T is added to the sourcerecognizer cancelled when its capture target dies
The guarantee

No relation edge ever survives the destruction of either endpoint, and every removal — including destroy-driven ones — is observable through the same two channels as every other primitive: the per-tick batch and the synchronous observer. Cascade chains terminate even on cyclic edges (ids never recycle; re-destroying is a no-op).

Relation reactivity#

Net-cancellation applies as everywhere else: opposing mutations of the same edge in one tick cancel the opposite buffer, and the last transition's buffer keeps the edge.

Resources#

Resources are singletons — world-level state that isn't attached to any entity. Viewport state, configuration, the current tool, or a class instance like a spatial index.

A resource store is created lazily on first access, seeded from its defaults. Like components, updates are partial and shallow-merged — and like everything else, writes are observable: setResource fires onResourceChanged(R, (prev, next)) synchronously and the type lands in the per-tick queryChangedResources() buffer.

  • Reads are silent. getResource creates the store on first use but emits nothing and never appears in queryChangedResources().
  • Registration follows store creation. getRegisteredResources() lists a type after its first getResource / setResource / onResourceChanged — defining alone doesn't register.
  • No removal, no built-in reset. Resources live for the world's lifetime; "reset" is just setResource(R, R.defaults) (remember it's a shallow merge — spread nested objects if you have them).

Cached queries#

query() returns the ids of every entity that has all of the given components and tags. Results are cached per type-combination and updated incrementally as components and tags are added or removed — repeat queries are cheap, with no per-call world scan.

Negation with Not()#

Wrap a component or tag in Not() to require its absence. Negated terms participate in the same incremental cache maintenance: adding the negated type to an entity evicts it from the cached result, removing the type re-admits it.

Note

A query needs at least one positive term (a component or a tag — relations and resources are not query terms) — a query of only Not() terms would have to scan every entity, so it throws instead. There is deliberately no Or operator: union two queries in user code if you need one. One cache exists per exact term signature (query(A, Not(B)) and query(A, B) are separate caches) and is retained for the world's lifetime — fine for the stable query sets real apps have, but don't generate query signatures dynamically.

Change tracking#

Beyond "who has X right now," the world answers "whose X changed this tick." Five batch queries read the per-tick buffers, which reset when you call clearDirty():

queryAddedTag(T)gained tag T this tick.

Net-cancellation#

Opposing mutations in the same tick cancel the opposite buffer: a remove clears the entity from queryAdded (and queryChanged); an add clears it from queryRemoved. The last transition wins, and its own buffer keeps the entry. The invariant the buffers guarantee is therefore about current state: an id in queryAdded has the component now; an id in queryRemoved does not. The same rule holds for tags and relation edges.

Note the consequence: relative to the start of the tick, that [e] is a removal of something the consumer never saw added. Make removal handling idempotent (removing an id that was never inserted is a no-op) and this is a non-issue — which the spatial-index example above already is.

Events vs buffers

This is the conceptual split to internalize: events are lossless and synchronous — every mutation fires, in order, as it happens. Per-tick buffers are coalesced — opposing mutations cancel, and what remains is consistent with where the world is now. Use events for journals, sync, and per-entity UI; use buffers for derived-state systems that only care where things ended up.

Removal includes destruction

queryRemoved and queryRemovedTag include entities torn down by destroyEntity. Consumers managing external resources (GPU buffers, DOM nodes, subscriptions) get a single channel for "this entity no longer has C" — no separate destroy bookkeeping.

Events#

This is the part most ECS libraries skip. Every mutation emits a synchronous event, and every subscription returns an Unsubscribe function.

Semantics worth knowing#

  • Synchronous. Handlers run inside the mutating call, in the order they subscribed. Per-entity handlers fire before wildcard handlers.
  • onComponentChanged fires on add too, with prev === undefined — one subscription covers both "appeared" and "updated."
  • onComponentRemoved fires before deletion. The store still holds the value, and prev hands it to you — the right moment to release anything keyed by it.
  • Per-entity filtering is built in. Pass an entityId as the last argument to onComponentChanged, onComponentRemoved, onTagAdded, or onTagRemoved.

Origin tagging#

Modules that both observe the world and mutate it — a sync adapter applying remote changes, an undo journal replaying inverse patches — must not re-observe their own writes, or you get echo loops and corrupted history. withOrigin tags every mutation made synchronously inside its callback; world.mutationOrigin exposes the tag to every handler that mutation fires.

  • Default is undefined — the implicit "local user" origin. The library defines no origin values; the vocabulary is yours.
  • Synchronous and exact. Events fire inside the mutating call, so a handler always sees the origin of exactly the mutation that fired it.
  • Re-entrant. Nested calls stack — innermost wins, restored on exit, exception-safe.
  • Cascades inherit. Everything destroyEntity triggers — including relation policy effects — carries the origin of the original call.
  • Synchronous window only. Mutations after an await are untagged; async continuations re-enter withOrigin.
What it is not

Not a transaction: there is no batching or deferred delivery — events stay synchronous and per-mutation, and the per-tick buffers remain origin-blind (derived-state systems must converge on world state no matter who wrote it).

Destroy semantics#

destroyEntity is not a silent teardown. It runs a precise, observable sequence:

  1. onEntityDestroyed listeners fire first — while the entity's components, tags, and relations are still fully readable, so callbacks can snapshot whatever they need.
  2. Query caches drop the entity.
  3. The relation sweep runs: every edge where the entity is source or target is removed — onRelationRemoved fires per edge (the dying entity's components are still readable from these handlers), ids land in the queryRelationRemoved buffer, and each relation's onTargetDestroy policy effect is collected, not yet applied.
  4. Each owned component is removed: onComponentRemoved fires with the dying value, and the id lands in the queryRemoved buffer.
  5. Each owned tag is removed: onTagRemoved fires, and the id lands in the queryRemovedTag buffer. Per-entity subscriptions are cleaned up.
  6. Deferred policy effects apply last — cascades destroy their sources, { tag } effects add their tags (skipped if a prior cascade already killed the source). Only now is mutation safe again.
Design rule

Destruction is just removal, observed through the same channels. If your cleanup logic handles onComponentRemoved / queryRemoved, it already handles destruction — there is no second code path to forget.

Don't mutate mid-destroy

Handlers fired during a destroy (onEntityDestroyed, onComponentRemoved, onTagRemoved, onRelationRemoved in steps 1–5) run while the world is mid-teardown: read freely, but do not mutate the world from them — create, destroy, relate, or add/remove anything. Mutate from a system reacting to the removed buffers on the next pass instead, or use a relation's onTargetDestroy policy, which the world applies for you at the safe point (step 6). The same handlers are safe to mutate from when fired by ordinary removals outside a destroy.

SystemScheduler#

Systems are named functions over the world, declared with defineSystem. SystemScheduler orders them by after / before constraints using a stable topological sort (Kahn's algorithm — registration order breaks ties), and runs them in that order each execute().

  • Re-registering a name replaces the system — handy for hot-reload.
  • Unknown constraint targets are tolerated — after: 'notRegistered' is ignored rather than an error, so systems can be registered in any order.
  • Cycles throw with the names involved.

PhasedScheduler#

Once a project grows past a handful of systems, a pure constraint graph stops describing intent — after: 'cull' works, but doesn't say why. PhasedScheduler adds named phases that you declare at construction time. The library ships zero phase opinions — it's purely the mechanism: bucketing, cross-phase validation, and profiler instrumentation.

Options#

FieldRequiredDescription
phasesyes Phase order, earliest first. Non-empty, no duplicates. Use as const on the array literal to get type-narrowed phase strings throughout the API.
defaultPhaseno Phase applied when register() receives a system without a phase. If unset, registering an unstamped system throws — phase membership becomes mandatory at the call site.

Rules#

  • Within a phase, after / before topo-sort exactly as in SystemScheduler (it runs one internally per phase).
  • Across phases, ordering is implicit in phase order — earlier phase always runs first.
  • Cross-phase constraints that contradict phase order are rejected at first execute() with an error that explains the fix. Redundant ones — an after pointing at an earlier phase — are tolerated, since phase order already satisfies them.
  • Phase membership is validated at register time: an unknown phase throws immediately.
  • Re-registering into a different phase moves the system — it never runs twice.

Pick a vocabulary that matches your domain#

Which scheduler?

SystemScheduler for small projects and custom pipelines. PhasedScheduler past ~6–8 systems, especially when observers and invariant-maintenance mix with derived-state computation — a phase gives the next contributor a one-line answer to "when does this run?"

Run conditions#

A system can declare a runIf predicate — evaluated immediately before it would run; return false to skip the tick. The library attaches no change-detection policy: the predicate is yours, and the per-tick buffers are the natural fuel for it.

Skips are observable: if the scheduler's profiler implements the optional skipSystem(name) hook, it fires instead of beginSystem/endSystem — which is how a devtools panel shows run/skip/idle stats per system.

Ordering caveat

Per-tick buffers are cleared at end of tick, so order systems that lazily read a type after the systems that write it (phases make this natural). A write that happens after the guard ran this tick is invisible to next tick's guard.

Profiler hook#

Both schedulers accept any object matching SystemProfiler on their profiler property. The scheduler knows nothing about measurement — it just calls the hooks if present.

beginPhase / endPhase are optional, so a profiler written for one scheduler works with the other unchanged.

Introspection#

For devtools, editors, and serialization, the world can enumerate everything it holds — because the library owns the stores, this works generically over any component shape.

All are O(k) in the number of registered types — no hidden scan of entity storage. Combine with onEntityCreated / onEntityDestroyed / onComponentChanged to build a live inspector, or walk getAllEntities() + getComponent() to write a serializer in a few dozen lines.

Saving & restoring#

Serialization stays in your code — walk the world with the introspection API and keep your own format (which state is "document" versus "transient" is an application opinion). Restoring is the part that needed library support: createEntityWithId recreates entities under their original ids, so saved entity references — including relation edges — are valid as-is. Nothing needs remapping.

  • Ascending order. createEntityWithId throws on ids below the counter, so restore sorts ids first — that one sort() is what keeps the never-reuse invariant unconditional.
  • Restore the counter too. Saved worlds may hold stale references to entities destroyed before the save; setNextEntityId keeps those ids dead forever instead of letting new entities silently inherit them.
Restore, not resurrection

These methods restore into a fresh world; they cannot re-create a destroyed id in a live one — that would break entityExists(staleId) === false forever. For undoable deletion in an editor, prefer a tombstone tag over destroyEntity, and reserve destruction for entities past the undo horizon.

Devtools#

The library ships inspection tooling out of the box, as two subpath exports that leave the core untouched: a headless lifecycle recorder (@jamesyong42/reactive-ecs/devtools, zero dependencies) and two React components (@jamesyong42/reactive-ecs/devtools/react, React is an optional peer dependency).

  • EcsInspector — a draggable, resizable, layout-persisting floating window: every entity with its tags and live component values, plus a timeline tab when a recorder is provided. Extend it with summary (header text) and tabs (custom panels).
  • EntityTimeline — a Chrome-tracing-style canvas waterfall of entity lifecycles: one bar per entity from birth to death, pan/zoom/live-tail, wall-clock or tick axis. Renders thousands of rows at 60fps.
  • createLifecycleRecorder — captures births and deaths from the event hooks (not a poll), so even entities born and destroyed within one tick are recorded. Because destroyEntity fires listeners before teardown, each record freezes the entity's final identity at death.

Identity is read off composition through one pluggable seam: an EntityDescriber returns { label, color?, detail?, outcome? } per entity. The default needs zero configuration (labels from component/tag names); pass your own to get domain labels, colors, and win/lose end-caps in the timeline.

Non-goals#

Knowing what a library refuses to do tells you as much as its feature list.

  • Archetype storage / SoA TypedArray packing. Component data lives in plain Map<EntityId, T> stores. Fine for thousands of entities with rich data; not built for millions of game entities scanned every frame.
  • Parallel / worker execution. Single-threaded by design — synchronous events and worker parallelism don't mix.
  • A serialization format. Walk the world with getAllEntities() / getComponentsOf() / getComponent() — which state is "document" versus "transient" is an application opinion, so the codec stays in your code.
Reaching for raw speed?

If your bottleneck is iterating huge homogeneous entity sets per frame, you've outgrown this library's trade-offs — use bitECS or becsy and hand-roll the reactivity you need. The two designs are genuinely different tools.
