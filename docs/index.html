A reactive ECS
built for UIs.

reactive-ecs is a dependency-free TypeScript Entity-Component-System designed for UI frameworks, editors, and tools — not just game loops. Component changes, tag flips, and resource updates all emit events you can subscribe to. Queries are cached and update incrementally. A topological scheduler keeps systems ordered.

zero dependencies typescript-first esm + cjs browser · node · bun · deno

Introduction#

Most ECS libraries are built for throughput: pack data flat, scan it fast, and leave change detection to you. That is the right trade for a game simulating 100k entities. It is the wrong trade for an editor, a canvas tool, or any app where a UI must react to world state.

reactive-ecs makes the opposite trade. The library owns component storage, and because it owns the store, it can make guarantees a bring-your-own-storage toolkit cannot:

  • Every write through the world API emits. addComponent / updateComponent / removeComponent all notify observers — no set() ceremony to forget. The flip side of the contract: reads hand you the stored object, so treat it as read-only and write through the API.
  • Subscriptions can target a single entity. onComponentChanged(Position, handler, entityId) is exactly the shape a useComponent() hook needs.
  • Removal is observable with the data still readable. onComponentRemoved fires before deletion, so handlers can release GPU buffers, DOM nodes, or subscriptions keyed by the dying value.
  • Entity ids never lie. Ids are monotonic and never reused — entityExists(staleId) is false forever.
  • Edges are managed. Relations keep an always-coherent inverse index, and no edge ever survives the destruction of either endpoint.
Positioning

If you want raw per-frame throughput for tens of thousands of game entities, use bitECS or becsy. If you want structured world state that a UI can subscribe to without reinventing the plumbing, this is for you. See non-goals for the full list of trade-offs.

Design principles#

Every API decision in this library answers to five rules:

  1. Absence is never silent. Reads return undefined; non-creating writes against absence throw.
  2. One ownership rule. Plain data crossing the API boundary is owned by the world: cloned in, frozen in dev, replaced never mutated. Class instances stay the caller's, by reference.
  3. Events are the journal; change detection is the partition. Events are lossless, synchronous, per-mutation. changes() classifies the net transition since the tick began into disjoint added / changed / removed sets, carrying values; the events remain the lossless capture layer every change set is a window over.
  4. Entity first. Every per-entity read/write is (entity, type, ...).
  5. The tick is one call — and the tick delivers. tickWorld is the sole frame-advance point: it seals, resets, delivers onChanges runs, and advances the clock. No second clock, no public flush/commit.

Installation#

Ships ESM and CJS with bundled type declarations. No runtime dependencies. Requires Node ≥ 20 when used server-side.

Quick start#

tickWorld is the whole frame protocol in one call — run systems, seal and reset the tick's change window, deliver onChanges runs, notify frame subscribers, advance the tick counter. Drive it from requestAnimationFrame, a setInterval, or only when something actually changed; the library has no opinion.

World & the tick#

A world is the container for all ECS state: entities, component stores, tag stores, resources, query caches, and subscriptions. Worlds are independent — create as many as you need.

Frame lifecycle#

The tick is one call, and it is the sole frame-advance point: tickWorld(world, (w) => scheduler.execute(w)) runs your callback, then — as one indivisible step — seals the open origin-run, resets the tick's change window, delivers the sealed runs to onChanges handlers, fires onFrame, and increments currentTick.

StepWhat it does
seal + resetFreezes the open run's values, then resets the changes() window and its partition baselines to fresh. Owns what clearDirty() used to.
deliverHands each sealed run to onChanges handlers, serially, in capture order. Delivery is after the reset, so a handler's own writes land in the next tick.
onFrameFires every onFrame subscriber — the post-delivery flush hook where a UI adapter pushes its batched state.
incrementTickAdvances currentTick by one.
Note

The frame always advances, even if an onChanges or onFrame handler throws — errors are collected and rethrown as an AggregateError after the clock has moved, so a buggy subscriber can't strand the world between frames. Call tickWorld(world) with no callback to flush and advance a frame without running systems — the "just deliver" path for an app with no scheduler loop.

Entities#

Entities are opaque integer ids (EntityId). Creation is O(1); destruction does work proportional to what the entity owns (components, tags, relation edges) plus the number of registered stores and query caches — fine at this library's scale, stated for honesty.

Ids are never reused#

Ids are allocated from a monotonic counter and never recycled. Once an entity is destroyed, its id is dead for the lifetime of the world.

Why this matters

Many ECS libraries recycle ids immediately, which means a stale id held by your UI, an undo stack, or a network message can silently alias a brand-new entity. Here a dangling reference is always detectably dangling: entityExists(staleId) returns false, never a lie. No generational indices required.

Components#

Components are typed data attached to entities. Define one with a name and a defaults object — the defaults double as the component's TypeScript shape.

Reading and writing#

  • addComponent is the upsert. It shallow-merges your data over the defined defaults (partial initialization is safe) and attach-or-replaces: re-adding replaces the value, with the existing value as prev in observers.
  • Freeze on every write. Managed plain data (arrays and {}-objects) is deep-frozen in place on the way in — immutable by construction, no defensive clone. Two entities can safely share a frozen subtree (it can only be replaced, never mutated), and a reference you keep to data you passed is now frozen too. Class instances and typed arrays are borrowed by reference. (Data must be acyclic and free of accessors.)
  • updateComponent is the functional update. It takes a recipe (prev) => next — p => ({ ...p, x }) is the one-level merge — and emits a change event. Returning prev unchanged is a free no-op. It never creates: absence is never silent, so it throws rather than no-op when the target is missing.

The contract

All writes go through addComponent / updateComponent / removeComponent, which is what lets the library guarantee that every change is queryable via change tracking and observable via events. One ownership rule covers both directions: managed plain data crossing the API boundary is owned by the world — frozen in place on the way in (immutable by construction, no defensive clone), and stored values are replaced on write, never mutated. Don't mutate the object returned by getComponent — the Readonly typing flags that at compile time, and the deep freeze (every build, not just dev) throws at runtime. Borrowed values (class instances, typed arrays) are by reference and outside the freeze — change them by replacement.

Tags#

Tags are zero-data boolean markers — presence is the information. Use them for states like Selected, Hovered, Dirty.

Tags participate in query() alongside components, have their own add/remove events, and their own changes() accessors (changes().addedTag(T) / changes().removedTag(T)).

Relations#

A relation is a managed, inverse-indexed, lifecycle-cleaned edge between two entities — for hierarchies, ownership, targeting, or any entity-to-entity reference you would otherwise store as a raw EntityId field and guard by hand.

Entity first, like every per-entity call: getSources(parent, ChildOf) reads "who points at parent?".

The inverse index is maintained by the store, never by you — no more materialized children arrays kept in sync by hand. The litmus for when to use a relation instead of a plain id field: make it a relation only if you query the inverse.

  • Born-dangling edges are impossible — relate throws if either endpoint is not alive.
  • Exclusivity replaces. On a sourceExclusive relation, relating to a new target unrelates the old one first — observable as removed-then-added, mirroring addComponent overwrite semantics.
  • Re-relating an existing edge is a no-op, mirroring addTag.

Destroy policies#

When a target dies, every edge pointing at it is removed and the relation's onTargetDestroy policy is applied to each source. Policy effects are deferred until the destroy completes, so handlers never observe a half-destroyed world:

PolicyEffect on each sourceTypical use
'clear' (default)none — the edge is dropped, the source lives onparent removed → child returns to root
'cascade'the source is destroyed tooselection chrome reaped with its target
{ tag: T }tag T is added to the sourcerecognizer cancelled when its capture target dies
The guarantee

No relation edge ever survives the destruction of either endpoint, and every removal — including destroy-driven ones — is observable through the same two channels as every other primitive: the per-tick batch and the synchronous observer. Cascade chains terminate even on cyclic edges (ids never recycle; re-destroying is a no-op).

Relation reactivity#

The partition rule applies per edge: the changes() window classifies each edge's net transition since the tick began. Edges have no changed accessor, so unrelate-then-re-relate of a pre-existing edge is a vacuous present→present and appears in neither addedRelation nor removedRelation.

Resources#

Resources are singletons — world-level state that isn't attached to any entity. Viewport state, configuration, the current tool, or a class instance like a spatial index.

A resource store is created lazily on first access, seeded from its defaults. Like components, updates are partial and shallow-merged — and like everything else, writes are observable: setResource fires onResourceChanged(R, (prev, next)) synchronously and the type appears in changes().changedResources() for the tick.

  • Reads are silent. getResource creates the store on first use but emits nothing and never appears in changes().changedResources().
  • Registration follows store creation. getRegisteredResources() lists a type after its first getResource / setResource / onResourceChanged — defining alone doesn't register.
  • No removal, no built-in reset. Resources live for the world's lifetime; "reset" is just setResource(R, R.defaults) (remember it's a shallow merge — spread nested objects if you have them).

Cached queries#

query() returns the ids of every entity that has all of the given components and tags. Results are cached per type-combination and updated incrementally as components and tags are added or removed — repeat queries are cheap, with no per-call world scan.

Negation with Not()#

Wrap a component or tag in Not() to require its absence. Negated terms participate in the same incremental cache maintenance: adding the negated type to an entity evicts it from the cached result, removing the type re-admits it.

Note

A query needs at least one positive term (a component or a tag — relations and resources are not query terms). The reason a pure-Not() query throws is maintenance, not the one-off scan: a no-arg query() is cheap because it copies the maintained alive set, but a pure-negative query would need its own cached set re-evaluated on every mutation of the negated types and on every entity create. It's that standing cost the throw refuses. There is deliberately no Or operator: union two queries in user code if you need one.

Disposing a cache entry#

One cache exists per exact term signature (query(A, Not(B)) and query(A, B) are separate caches). Entries live until you drop them: disposeQuery(...types) removes the cache entry and its reverse-index registrations for that exact signature (order-insensitive, like query). Re-querying rebuilds the entry with one scan over the smallest candidate store — that rebuild is the only full-scan path in the library — and resumes incremental maintenance.

Change tracking#

Beyond "who has X right now," the world answers "whose X changed this tick, and to what." changes() is that answer — a value-carrying view of every change since the tick began, governed by one invariant, the partition rule: each accessor classifies an entity's net transition since the tick began into disjoint added / changed / removed sets, now carrying the values. The accessor verbs mirror the partition exactly:

AccessorNet transition since the tick began · value
changes().added(C)absent → present. Map<id, value> — the attached value.
changes().changed(C)present → present with ≥ 1 write. Map<id, {prev, next}>. A fresh attach is never here.
changes().removed(C)present → absent — via removeComponent or destroyEntity. Map<id, value> — the dying value.
changes().addedTag(T) / removedTag(T)same partition for tags — no changed set; present → present is vacuous.
changes().addedRelation(R) / removedRelation(R)same partition per edge — no changed set.
changes().changedResources()resource types written via setResource this tick, {prev, next} each.

Migration

The presence-only buffer queries — queryAdded / queryChanged / queryRemoved and their tag / relation / resource twins — were removed in v0.16. Each maps to the value-carrying accessor of the same name on changes(): queryAdded(C) → changes().added(C). The old queries returned ids only; changes() returns ids and values.

The partition rule#

Within a tick, only the net transition matters. absent→present is added only; present→present with at least one write is changed only; present→absent is removed only; absent→absent lands nowhere. Consequences:

  • A fresh attach is in added only — "new" and "updated" never alias.
  • Remove-then-re-add of a component that existed at tick start nets to changed — the world holds it before and after.
  • Add-then-remove (or create + add + destroy) within one tick lands in no set.
  • destroyEntity participates: pre-existing components/tags/edges land in removed with their dying value; same-tick ephemera land nowhere.

Two channels: react vs. record#

The library carries change in two forms, and one sentence tells you which to reach for. Synchronous events are for reacting — invariants, GPU/DOM cleanup, fine-grained value subscriptions: things that need the moment, mid-mutation. onChanges delivered runs are for recording — undo, sync, derived state, devtools: things that want the net result of a coherent unit of work, with attribution, after the work is done.

A delivered run is a changes() view — same added / changed / removed accessors — plus the run's origin, frozen at seal time and safe to retain across thousands of ticks. That, plus applyChanges (which replays a run or its inverse through the public write API), is the whole ground support: app-level undo/redo lands in about fifteen lines — push runs on onChanges, and applyChanges(world, c, { origin: HISTORY, invert: true }) to undo. See the tombstone doctrine for why ids stay valid across delete → undo → redo.

Removal includes destruction

changes().removed(C) and removedTag(T) include entities torn down by destroyEntity. Consumers managing external resources (GPU buffers, DOM nodes, subscriptions) get a single channel for "this entity no longer has C" — no separate destroy bookkeeping.

Events#

This is the part most ECS libraries skip. Every mutation emits a synchronous event, and every subscription returns an Unsubscribe function.

Semantics worth knowing#

  • Synchronous. Handlers run inside the mutating call, in the order they subscribed. Per-entity handlers fire before wildcard handlers.
  • onComponentChanged fires on add too, with prev === undefined — one subscription covers both "appeared" and "updated."
  • onComponentRemoved fires before deletion. The store still holds the value, and prev hands it to you — the right moment to release anything keyed by it.
  • Per-entity filtering is built in. Pass an entityId as the last argument to onComponentChanged, onComponentRemoved, onTagAdded, or onTagRemoved.
  • onChanges is the one delivered channel. It does not fire mid-mutation — it delivers one net, origin-tagged change set per origin-run at the tick (see Two channels). Reach for the synchronous events to react; reach for onChanges to record.

Origin tagging#

Modules that both observe the world and mutate it — a sync adapter applying remote changes, an undo journal replaying inverse patches — must not re-observe their own writes, or you get echo loops and corrupted history. withOrigin tags every mutation made synchronously inside its callback; world.mutationOrigin exposes the tag to every handler that mutation fires.

  • Default is undefined — the implicit "local user" origin. The library defines no origin values; the vocabulary is yours.
  • Synchronous and exact. Events fire inside the mutating call, so a handler always sees the origin of exactly the mutation that fired it.
  • Re-entrant. Nested calls stack — innermost wins, restored on exit, exception-safe.
  • Cascades inherit. Everything destroyEntity triggers — including relation policy effects — carries the origin of the original call.
  • Synchronous window only. Mutations after an await are untagged; async continuations re-enter withOrigin.
What it is not

Not a transaction: there is no batching or deferred delivery — events stay synchronous and per-mutation, and the per-tick buffers remain origin-blind (derived-state systems must converge on world state no matter who wrote it).

Destroy semantics#

destroyEntity is not a silent teardown. It runs a precise, observable sequence:

  1. onEntityDestroyed listeners fire first — while the entity's components, tags, and relations are still fully readable, so callbacks can snapshot whatever they need.
  2. Query caches drop the entity.
  3. The relation sweep runs: every edge where the entity is source or target is removed — onRelationRemoved fires per edge (the dying entity's components are still readable from these handlers), the edges appear in changes().removedRelation(R), and each relation's onTargetDestroy policy effect is collected, not yet applied.
  4. Each owned component is removed: onComponentRemoved fires with the dying value, and the id appears in changes().removed(C) (mapped to its dying value).
  5. Each owned tag is removed: onTagRemoved fires, and the id appears in changes().removedTag(T). Per-entity subscriptions are cleaned up.
  6. Deferred policy effects apply last — cascades destroy their sources, { tag } effects add their tags (skipped if a prior cascade already killed the source). Only now is mutation safe again.
Design rule

Destruction is just removal, observed through the same channels. If your cleanup logic handles onComponentRemoved / changes().removed(C), it already handles destruction — there is no second code path to forget.

Mutation is rejected mid-destroy

Handlers fired during a destroy (onEntityDestroyed, onComponentRemoved, onTagRemoved, onRelationRemoved in steps 1–5) run while the world is mid-teardown: read freely, but mutating the world from them throws — cannot mutate the world from a handler during entity teardown — react via the removed buffers or an onTargetDestroy policy instead. React from a system reading the removed buffers on the next pass, or use a relation's onTargetDestroy policy, which the world applies for you at the safe point (step 6). The guard covers only the destroy sweep — the same handlers can mutate freely when fired by ordinary removals outside a destroy.

Document vs. interaction state#

In an editor-shaped app the world holds two kinds of entities, and they want different deletion semantics. This is architecture, not a workaround:

  • Interaction entities — gesture recognizers, pointers, selection chrome, drag ghosts. They churn, lose arbitrations, and die. Delete them with destroyEntity and let cascade / { tag } policies tear down their dependents. Their ids appearing in saved data would be a bug.
  • Document entities — shapes, layers, notes: anything inside the undo horizon. "Deleting" one must be reversible, and ids are never reused, so destroyEntity is the wrong verb while undo can still resurrect it. Use a tombstone tag and exclude it from document queries.

destroyEntity for a document entity then means exactly one thing: eviction past the undo horizon — the moment the app guarantees no undo stack, clipboard, or collaborator can ever reference that id again.

SystemScheduler#

Systems are named functions over the world, declared with defineSystem. SystemScheduler orders them by after / before constraints using a stable topological sort (Kahn's algorithm — registration order breaks ties), and runs them in that order each execute().

  • Re-registering a name replaces the system — handy for hot-reload.
  • Unknown constraint targets throw at first execute, naming both parties — a typo must not silently reorder the pipeline. Validation is deferred to execute() (and re-runs after every register/remove), so systems can still be registered in any order.
  • Cycles throw with the names involved.
Optional dependencies

There is deliberately no "ignore if missing" escape: optional-dependency ordering across app configurations is what phases are for — phase: 'derive' says "after all simulation, whatever simulation is installed" without naming any particular system.

PhasedScheduler#

Once a project grows past a handful of systems, a pure constraint graph stops describing intent — after: 'cull' works, but doesn't say why. PhasedScheduler adds named phases that you declare at construction time. The library ships zero phase opinions — it's purely the mechanism: bucketing, cross-phase validation, and profiler instrumentation.

Options#

FieldRequiredDescription
phasesyes Phase order, earliest first. Non-empty, no duplicates. Use as const on the array literal to get type-narrowed phase strings throughout the API.
defaultPhaseno Phase applied when register() receives a system without a phase. If unset, registering an unstamped system throws — phase membership becomes mandatory at the call site.

Rules#

  • Within a phase, after / before topo-sort exactly as in SystemScheduler (it runs one internally per phase).
  • Across phases, ordering is implicit in phase order — earlier phase always runs first.
  • Cross-phase constraints that contradict phase order are rejected at first execute() with an error that explains the fix. Redundant ones — an after pointing at an earlier phase — are tolerated, since phase order already satisfies them.
  • Unknown constraint targets throw at first execute(), naming both parties — same rule as SystemScheduler, validated against the full cross-phase registry.
  • Phase membership is validated at register time: an unknown phase throws immediately.
  • Re-registering into a different phase moves the system — it never runs twice.

Pick a vocabulary that matches your domain#

Which scheduler?

SystemScheduler for small projects and custom pipelines. PhasedScheduler past ~6–8 systems, especially when observers and invariant-maintenance mix with derived-state computation — a phase gives the next contributor a one-line answer to "when does this run?"

Run conditions#

A system can declare a runIf predicate — evaluated immediately before it would run; return false to skip the tick. The library attaches no change-detection policy: the predicate is yours, and the changes() window is the natural fuel for it.

Skips are observable: if the scheduler's profiler implements the optional skipSystem(name) hook, it fires instead of beginSystem/endSystem — which is how a devtools panel shows run/skip/idle stats per system.

Ordering caveat

Per-tick buffers are cleared at end of tick, so order systems that lazily read a type after the systems that write it (phases make this natural). A write that happens after the guard ran this tick is invisible to next tick's guard.

Profiler hook#

Both schedulers accept any object matching SystemProfiler on their profiler property. The scheduler knows nothing about measurement — it just calls the hooks if present.

beginPhase / endPhase are optional, so a profiler written for one scheduler works with the other unchanged.

Introspection#

For devtools, editors, and serialization, the world can enumerate everything it holds — because the library owns the stores, this works generically over any component shape.

All are O(k) in the number of registered types — no hidden scan of entity storage. Combine with onEntityCreated / onEntityDestroyed / onComponentChanged to build a live inspector, or walk getAllEntities() + getComponent() to write a serializer in a few dozen lines.

Saving & restoring#

Serialization stays in your code — walk the world with the introspection API and keep your own format (which state is "document" versus "transient" is an application opinion). Restoring is the part that needed library support: createEntityWithId recreates entities under their original ids, so saved entity references — including relation edges — are valid as-is. Nothing needs remapping.

  • Ascending order. createEntityWithId throws on ids below the counter, so restore sorts ids first — that one sort() is what keeps the never-reuse invariant unconditional.
  • Restore the counter too. Saved worlds may hold stale references to entities destroyed before the save; setNextEntityId keeps those ids dead forever instead of letting new entities silently inherit them.
Restore, not resurrection

These methods restore into a fresh world; they cannot re-create a destroyed id in a live one — that would break entityExists(staleId) === false forever. For undoable deletion in an editor, prefer a tombstone tag over destroyEntity, and reserve destruction for entities past the undo horizon — see document vs. interaction state.

Devtools#

The library ships inspection tooling out of the box, as two subpath exports that leave the core untouched: a headless lifecycle recorder (@jamesyong42/reactive-ecs/devtools, zero dependencies) and two React components (@jamesyong42/reactive-ecs/devtools/react, React is an optional peer dependency).

  • EcsInspector — a draggable, resizable, layout-persisting floating window: every entity with its tags and live component values, plus a timeline tab when a recorder is provided. Extend it with summary (header text) and tabs (custom panels).
  • EntityTimeline — a Chrome-tracing-style canvas waterfall of entity lifecycles: one bar per entity from birth to death, pan/zoom/live-tail, wall-clock or tick axis. Renders thousands of rows at 60fps.
  • createLifecycleRecorder — captures births and deaths from the event hooks (not a poll), so even entities born and destroyed within one tick are recorded. Because destroyEntity fires listeners before teardown, each record freezes the entity's final identity at death.

Identity is read off composition through one pluggable seam: an EntityDescriber returns { label, color?, detail?, outcome? } per entity. The default needs zero configuration (labels from component/tag names); pass your own to get domain labels, colors, and win/lose end-caps in the timeline.

Non-goals#

Knowing what a library refuses to do tells you as much as its feature list.

  • Archetype storage / SoA TypedArray packing. Component data lives in plain Map<EntityId, T> stores. Fine for thousands of entities with rich data; not built for millions of game entities scanned every frame.
  • Parallel / worker execution. Single-threaded by design — synchronous events and worker parallelism don't mix.
  • A serialization format. Walk the world with getAllEntities() / getComponentsOf() / getComponent() — which state is "document" versus "transient" is an application opinion, so the codec stays in your code.
Reaching for raw speed?

If your bottleneck is iterating huge homogeneous entity sets per frame, you've outgrown this library's trade-offs — use bitECS or becsy and hand-roll the reactivity you need. The two designs are genuinely different tools.
